A structured ecosystem of installable domain packages for building, routing, and operating high-value AI workflows at scale.

This repository is not a loose collection of prompts, notes, or experimental snippets. It is a practical package system designed for real use: domain-oriented, linked, maintainable, and ready to be downloaded, installed, and integrated into actual agent workflows.

Each domain package is built as a routed system rather than a flat archive. Instead of forcing an agent or operator to load everything at once, the package structure is designed to keep the entry point lightweight, preserve internal connectivity, and expose the right mini-skills, references, examples, and support files only when they are needed.

## Why this repository exists

Large skill ecosystems often fail for predictable reasons:

- too many disconnected files,
- weak or broken internal linking,
- duplicated or stale content,
- mixed formatting and inconsistent structure,
- isolated skills that ignore adjacent knowledge,
- packages that are theoretically interesting but operationally unusable.

This repository is built to solve those problems.

The goal is to provide domain packages that are:

- **practical**
- **installable**
- **internally linked**
- **English-only**
- **routing-friendly**
- **maintainable at scale**
- **usable in real execution environments**

These packages are designed for real work, not just for demonstration.

## What you get

Each domain package is a structured folder that typically includes:

- a lightweight `SKILL.md` router,
- a composition protocol,
- a self-diagnostic protocol,
- a catalog of mini-skills,
- a routing guide,
- an asset link index,
- a machine-readable manifest,
- mini-skill reference files,
- mini-skill example files,
- optional linked assets, scripts, templates, or support materials.

The system is designed to support selective loading, internal consistency, and scalable package maintenance.

## The repository is organized in five planes

This repository is structured across five practical planes:

### 1. AI and Agents

## Core design philosophy

The architecture follows a few strict ideas:

### Domain-first structure
Work should be grouped by real domains, not by random file accumulation.

### Mini-skills, not monoliths
Each domain is composed of connected mini-skills. These are separated for routing and selective loading, not because real work happens in isolation.

### Lightweight router, deep linked internals
The main router should remain small. Deep guidance belongs in references, examples, resources, and linked support files.

### Strict internal linking
Everything declared must exist. Everything linked must resolve. Package integrity is part of package usability.

### English-only user-facing content
All final package content is normalized to English for clarity, consistency, and reliable use.

### Practical maintainability
Packages must remain structurally coherent and small enough to maintain. File count, duplication, stale links, and broken records are treated as system problems, not cosmetic problems.

## Installation and usage

Each domain package is intended to be usable as a standalone routed package.

Typical usage flow:

1. Choose the relevant domain package.
2. Install or copy it into your working environment.
3. Start from the package `SKILL.md`.
4. Use the router to activate the correct mini-skills.
5. Load references, examples, and linked assets only as needed.
6. Use the self-diagnostic and composition protocols where applicable.

Specific installation or runtime steps may differ by package.
