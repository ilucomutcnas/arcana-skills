# Style Guardrails

## Non-negotiables
- Semantic tokens before raw values.
- Consistent naming across design and engineering.
- Variants before forks.
- States must be explicit: default, hover, active, focus, disabled, loading, selected, invalid, success, warning, destructive when relevant.
- Accessibility must survive theming changes.
- Motion must respect reduced-motion preferences.
- Every token should have a reason to exist.
- Every component API should be smaller than the number of edge cases it prevents.

## Tone of the system
A design system should feel:
- consistent without being sterile
- flexible without being vague
- scalable without becoming abstract sludge

## Default quality bar
A design-system output is not complete unless it is:
- reusable
- documented
- implementation-aware
- accessibility-conscious
- reviewable by another team
